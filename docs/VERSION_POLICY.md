# Version Policy

영중폼은 Semantic Versioning을 기준으로 버전을 관리한다.

```text
MAJOR.MINOR.PATCH
```

## 버전 의미

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| MAJOR | 운영 흐름이나 데이터 구조에 큰 변화가 있는 릴리즈 | `v2.0.0` |
| MINOR | 기존 호환성을 유지하는 기능 추가 | `v1.1.0` |
| PATCH | 버그 수정, 권한 수정, 작은 UX 개선 | `v1.0.1` |

## 운영 시작 기준

### v1.0.0

운영 릴리즈 기준 버전이다. 설문 생성, 응답 제출, 응답 관리, 결과보고서, 다운로드, 권한 체계가 운영 가능한 상태를 의미한다.

### v1.0.1

버그 수정 버전이다. 기존 기능의 의도와 UI를 바꾸지 않고 오류만 수정한다.

### v1.0.2

권한 수정 버전이다. Firestore rules, role 정규화, query 권한 불일치 등을 수정한다. 권한 확대가 포함되면 별도 승인과 QA가 필요하다.

### v1.0.3

UX 개선 버전이다. 기존 기능을 유지하면서 안내 문구, 메뉴 활성화, 오류 메시지 등 작은 사용성 개선을 반영한다.

### v1.1.0

기능 추가 버전이다. 새 설문 유형, 새 보고서 기능, 새 관리 도구 등 사용자 기능이 추가될 때 사용한다.

## Patch 버전 증가 기준

다음 변경은 PATCH를 올린다.

- permission-denied 수정
- 메뉴 활성화 오류 수정
- 빌드 오류 수정
- 운영 문구 오류 수정
- 보고서 저장 오류 수정
- 다운로드 실패 수정
- 특정 role 접근 오류 수정

## Minor 버전 증가 기준

다음 변경은 MINOR를 올린다.

- 새 설문 템플릿 기능
- 새 통계 화면
- 새 다운로드 형식
- 새 보고서 섹션
- 새 관리자 도구

## Major 버전 증가 기준

다음 변경은 MAJOR를 검토한다.

- responses 저장 구조 변경
- surveys 핵심 구조 변경
- 권한 모델 재설계
- 응답 제출 트랜잭션 변경
- quota 구조 변경
- 기존 URL 체계 변경

## 버전 태그 원칙

- 운영 배포가 완료된 커밋에만 버전 태그를 붙인다.
- dry-run 또는 QA 중인 커밋에는 운영 버전 태그를 붙이지 않는다.
- hotfix는 가장 가까운 운영 버전의 PATCH로 관리한다.
- 버전 변경에는 QA 결과와 배포 시간을 함께 기록한다.

## 운영 버전 라인

| 버전 라인 | 목적 | 포함 범위 |
| --- | --- | --- |
| `v1.0.x` | 운영 안정화 | permission-denied, 권한, 보고서, quota, 최근응답 안정화 |
| `v1.1.x` | 운영 기능 추가 | 운영 호환성을 유지하는 기능 추가 |
| `v1.2.x` | UX 개선 | 메뉴, 안내 문구, 화면 흐름, 관리 편의 개선 |
| `v2.0.0` | 구조 전환 | Draft-Publish 정식화, Versioning, 큰 데이터 구조 변경 |

## 기능별 버전 판단

- permission-denied 수정은 `v1.0.x` PATCH로 처리한다.
- quota count 안정화는 `v1.0.x` PATCH로 처리한다.
- creator/staff/viewer 권한 보정은 `v1.0.x` PATCH로 처리한다.
- 결과보고서 저장 오류 수정은 `v1.0.x` PATCH로 처리한다.
- 새 결과보고서 유형 추가는 `v1.1.x` MINOR로 처리한다.
- 관리 화면 사용성 개선은 `v1.2.x`로 분리할 수 있다.
- Draft-Publish와 Versioning이 운영 구조를 바꾸면 `v2.0.0`으로 검토한다.

## 운영 태그 기록 항목

버전 태그를 남길 때 다음을 함께 기록한다.

- 배포 일시
- 배포자
- 변경 요약
- Firestore rules 변경 여부
- 데이터 구조 변경 여부
- QA 체크리스트 결과
- smoke test 결과
- rollback 기준 commit
