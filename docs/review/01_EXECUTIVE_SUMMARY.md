# 01. Executive Summary — 영중 설문 플랫폼 v1.0 종합 진단

> 작성일: 2026-07-04
> 대상: 영중종합사회복지관 설문·신청·응답 관리 플랫폼 (React 19 + Vite 7 + Firebase Firestore)
> 성격: 기능 개발이 아닌 **전체 리뷰 및 v2.0 재설계 기준 문서**

## 1. 한 줄 결론

현재 플랫폼은 **"소수 전문 운영자가 조심스럽게 다루면 동작하는 정교한 단일 앱"** 이다.
설문 생성·응답·통계·보고서·다운로드까지 실제 복지관 실무 흐름을 이미 한 번에 커버하고 있다는 점은 강점이지만,
**장기간·다수 직원·다양한 조사로 확장할 때 무너지는 구조적 리스크가 명확히 존재**한다.

핵심 리스크는 다음 4가지에 집중되어 있다.

| # | 리스크 | 위험도 | 근거 |
|---|--------|--------|------|
| R1 | 공개 응답 엔드포인트에 App Check·rate limit·응답 크기 제한이 없음 | **Critical** | 익명 대량 제출로 Firestore 비용 폭증·정원 조작·데이터 오염 가능 |
| R2 | 권한 판정 로직이 클라이언트(`users.js`)와 규칙(`firestore.rules`) 두 곳에 중복되고, legacy 필드 호환이 과도 | **High** | 한쪽만 수정 시 permission-denied 또는 권한 상승. KI-001·003·004가 모두 이 원인 |
| R3 | Draft-Publish 분리 미완성 (KI-005). 저장 즉시 공개 설문이 바뀜 | **High** | 운영 중 설문 편집이 응답자 화면을 실시간 파괴 |
| R4 | 자동 백업·복구 체계 부재. 소프트 삭제는 있으나 문서 오염/실수 복구 수단이 제한적 | **High** | 운영자 실수 시 응답 원본 복구 불가 구간 존재 |

## 2. 8대 평가 기준 스코어카드

| 기준 | 점수 | 요약 |
|------|:----:|------|
| 1. 현장 직원이 쉽게 설문을 만들 수 있는가 | ★★★☆☆ | 4종 폼타입·템플릿·질문블록 등 구조는 우수하나 빌더가 2,600줄 단일 파일로 학습·오조작 리스크 큼 |
| 2. 주민이 쉽게 응답할 수 있는가 | ★★★★☆ | 모바일 반응형·자동임시저장·분기·페이지 모드 등 응답 UX 성숙. 접근성(스크린리더/라벨)은 미흡 |
| 3. 응답 데이터가 안전하게 보호되는가 | ★★☆☆☆ | 마스킹·익명화·소프트삭제는 있으나 공개 write 무방비, PII 평문 저장, App Check 부재 |
| 4. 권한 구조가 명확한가 | ★★☆☆☆ | role/status 판정이 이중화·legacy 과다로 "명확"보다 "누적 방어" 상태 |
| 5. 통계·보고서가 실무에 도움 되는가 | ★★★★☆ | 규칙기반 요약·자유응답 분류·Excel/DOCX 자동화까지 실무 지향. 다만 룰기반 한계 |
| 6. 다양한 조사로 확장 가능한가 | ★★★☆☆ | 폼타입·질문유형 확장성은 있으나 스키마 버전 필드 부재로 마이그레이션 취약 |
| 7. Firestore 구조가 안정적인가 | ★★★☆☆ | 컬렉션 분리는 합리적이나 응답 문서 비대·quota 이중 카운터·인덱스 최소화 |
| 8. 운영자가 실수해도 복구 가능한가 | ★★☆☆☆ | 소프트삭제/복원 부분 존재, 그러나 전체 백업·감사기반 롤백 없음 |

**종합: 2.9 / 5.0** — "실사용 가능하지만, 운영 규모 확대 전 구조 보강이 필수."

## 3. 강점 (유지·계승할 것)

- **실무 흐름 완결성**: 설문생성 → 게시 → 응답 → 통계 → 보고서 → CSV/Excel/DOCX 까지 한 플랫폼에서 끝난다. 복지관 현장에는 큰 가치.
- **정원(quota) 관리의 트랜잭션 처리**: 권역×연령대 매트릭스, 옵션별 정원, 1인 1슬롯, 중복방지 lock을 Firestore 트랜잭션으로 원자 처리. 신청형 폼의 핵심 자산.
- **개인정보 대응의 흔적**: 화면 마스킹(`privacy.js`), 응답 익명화(`anonymizeResponsePii`), 다운로드 마스킹이 이미 구현.
- **운영 문서화 문화**: `docs/`·`ai/` 아래 아키텍처·변경영향·알려진이슈 문서가 존재. 인수인계·회귀방지 기반이 이미 있음.

## 4. 가장 시급한 5가지 (Top 5 Action)

1. **[Critical] Firebase App Check + 공개 write 하드닝** — 응답 크기 상한, 필드 화이트리스트, App Check 토큰 검증. (06번 문서)
2. **[High] 권한 단일 소스화** — Custom Claims 기반으로 role/status를 토큰에 심고, 규칙·클라이언트가 동일 소스를 읽게. (06·11번 문서)
3. **[High] Draft-Publish 정식 분리** — `draftQuestions`(편집) vs `questions`(운영) 완전 분리 + 명시적 게시 액션. (05·13번 문서)
4. **[High] 자동 백업·복구 파이프라인** — 일일 Firestore export(GCS) + 응답 하드삭제 금지 유지 + 감사기반 롤백. (10·14번 문서)
5. **[Medium] `surveys.js`(4,483줄) 리포지토리 분리** — 응답/감사/quota/draft 도메인 모듈로 분해해 회귀 위험 감소. (09번 문서)

## 5. v2.0 방향성 선언

> 영중 설문 플랫폼 v2.0 = **현장 사용성 중심 · 안전한 응답데이터 구조 · 보고서 자동화 기반**

- **현장 사용성**: 빌더를 "질문 추가 위저드 + 실시간 미리보기"로 단순화, 폼타입 선택을 첫 화면 결정으로.
- **안전한 데이터**: App Check + Custom Claims + 서버측(Cloud Functions) 응답 검증 + 자동 백업. PII는 별도 서브컬렉션·접근 최소화.
- **보고서 자동화**: 규칙기반 요약을 유지하되, 통계 스냅샷을 응답 시점에 집계(집계 문서화)하여 대량 응답에서도 O(1) 보고서 생성.

각 축의 상세 설계는 `11_V2_MASTER_BLUEPRINT.md`, 이행 순서는 `13_IMPLEMENTATION_ROADMAP.md`를 따른다.
