## 4.1 설문 생성 UX (직원 관점)

### 현재 상태
- `SurveyBuilderPage.jsx` 단일 화면(2,621줄)에서 폼타입 선택 → 질문/섹션 편집(`SectionEditor`, `QuestionEditor`) → 미리보기 → 저장.
- 완성 템플릿(`formTemplates.js` — 실제 복지관 서식 다수 내장)과 질문 블록 프리셋(`QuestionBlockPicker`) 제공. **이건 강점** — 백지 공포를 줄여줌.

### 문제점
- **한 화면 과부하**: 폼타입·기능플래그(정원/중복/명단/처리상태)·질문편집·분기설정·일정·정원매트릭스가 한 페이지에 공존. 처음 쓰는 직원이 "어디부터 손대야 하나"를 판단하기 어렵다.
- **위험 설정과 일반 설정이 같은 층위**: 정원 매트릭스·중복방지처럼 잘못 만지면 응답을 막는 설정이, 제목 입력과 같은 화면에 노출.
- **저장 실패의 전부-아니면-전무**: payload 생성·검증이 한 컴포넌트에 있어 일부 오류가 저장 전체를 막음. 자동 임시저장(빌더측)이 없어 작성 중 이탈 시 유실 위험.
- **미리보기와 편집의 왕복**: 미리보기가 별도 라우트(`/preview`)라 편집↔확인 컨텍스트 전환 비용.

### 위험도: **High**
### 개선 방향
1. **위저드 3~4단계**: ① 유형 선택 → ② 문항 구성 → ③ 옵션(정원/중복/일정) → ④ 검토·게시. 각 단계에서만 관련 설정 노출.
2. **실시간 사이드 미리보기**: 편집 옆에 응답자 화면 라이브 렌더(별도 라우트 제거).
3. **빌더 자동저장**: draft 문서에 편집본 주기 저장(2.4의 draftQuestions와 연계).
4. **위험 설정 게이팅**: 정원·중복방지는 "고급 설정" 접힘 영역 + 변경 시 영향 안내.

### 우선순위: P1 / 난이도: 중~높음

## 4.2 수정 · 복제 UX

### 현재 상태
- 수정은 생성과 같은 빌더 재사용. 복제는 목록에서 `duplicateSurvey`.

### 문제점
- **게시된 설문 수정의 위험이 UI에 안 보임**: 응답이 이미 쌓인 설문을 편집할 때 "이 변경이 공개 화면·기존 응답 통계에 즉시 반영된다"는 경고가 없다(KI-005의 UX면).
- 복제 시 무엇이 복제/초기화되는지(응답 0, 정원 초기화) 사용자에게 명시되지 않음.

### 위험도: High
### 개선 방향: 게시 후 편집은 "새 버전으로 편집" 모드 강제 + 변경 diff 요약 + 명시적 재게시. 복제는 "복제 항목/제외 항목" 요약 표시.
### 우선순위: P1 / 난이도: 중

## 4.3 응답자 화면 UX (주민 관점) — **상대적 강점**

### 현재 상태 (SurveyResponsePage.jsx, 2,353줄)
- **자동 임시저장**: localStorage(`draft_{surveyId}_{userId}`) + 서버 draft(`draftResponses`), 이어서 작성 선택 UI.
- **분기(branching)**: `buildVisibleQuestionFlow`로 조건부 문항 흐름 계산, 미방문 문항 제출 차단.
- **페이지 모드 vs 단일 모드**: `responseMode`로 섹션 단위 페이지네이션 지원.
- **제출 멱등성**: `clientSubmitId` + sessionStorage로 중복 제출 방지, 완료 메시지 커스터마이즈.
- **정원 마감·마감시각·비공개** 상태별 안내 메시지 분기.

### 문제점
- **필수/오류 피드백의 시각적 강도**: `fieldErrors` 상태는 있으나 스크린리더 연결(`aria-invalid`, `aria-describedby`)이 없어 접근성 사용자에게 오류가 전달되지 않음(08 참조).
- **진행률·남은 문항 표시**: 페이지 모드에서 "몇 개 중 몇 번째"의 명시적 진행 인디케이터가 약함(고령 응답자 이탈 요인).
- **프로덕션 debug 로그**: `console.debug('[SurveyResponsePage] submit payload', {...})`가 응답 전체 payload(잠재 PII 포함)를 콘솔에 출력. DEV 가드가 있는지 확인 필요 — 있더라도 PII를 debug에 싣는 패턴은 위험.
- 긴 설문에서 27개 useState로 상태가 분산 → 렌더 성능·유지보수 부담(09 참조).

### 위험도: Medium (UX)·High (debug PII 노출 가능성)
### 개선 방향
- 오류를 인라인+요약(상단 "N개 항목 확인 필요")으로, ARIA 연결.
- 페이지 모드에 진행바 + "이전/다음" 고정 하단 바(모바일 엄지 도달 영역).
- debug payload 로깅에서 PII 필드 제거 또는 DEV 전용·마스킹.

### 우선순위: P1(디버그·오류접근성) / 난이도: 낮~중

## 4.4 모바일 접근성 (주민 대다수의 실제 환경)

### 현재 상태
- `index.html`에 `viewport` 메타 정상(`width=device-width, initial-scale=1.0`).
- `styles.css`에 미디어쿼리 3종(`960/640/600px`) + `print`. 응답 섹션 버튼에 `:focus-visible`.

### 문제점
- **미디어쿼리 개수가 적음**: 3,477줄 CSS에 브레이크포인트 3개 → 관리자 화면(테이블·정원 매트릭스·통계)은 좁은 화면에서 가로 스크롤 의존(`-webkit-overflow-scrolling: touch` 흔적). 직원이 현장에서 태블릿/폰으로 관리할 때 불편.
- **터치 타깃 크기 미검증**: 선택지·라디오·체크박스의 최소 44×44px 보장이 CSS에 명시되지 않음. 고령 응답자 오터치 유발.
- **폰트 하한**: `font-size: 0.78rem` 등 12~13px대 텍스트 다수. 고령 사용자 가독성 저하.

### 위험도: **Medium** (응답자) / High (직원 관리화면 모바일)
### 개선 방향: 응답자 화면 우선 — 최소 16px 본문, 44px 터치 타깃, 하단 고정 액션바. 관리화면은 태블릿 반응형 우선순위 지정.
### 우선순위: P2 / 난이도: 중

## 4.5 통계 화면 사용성 (실무 보고서 관점)

### 현재 상태
- `SurveyResponsesAdminPage`(2,889줄)에 통계·필터·CSV·명단·슬롯 다운로드 통합. `buildSurveyAnalytics` 기반 요약·자유응답 분류·프로그램명 정규화 집계.

### 문제점
- 통계·응답목록·다운로드·처리상태 관리가 한 화면에 밀집 → 보고서 작성자가 원하는 지표를 찾기까지 스크롤·필터 조작이 많음.
- 그래프/차트가 아닌 텍스트·표 중심 → 보고서에 바로 붙일 시각자료 생성이 약함(현재 DOCX는 텍스트 위주).

### 위험도: Medium
### 개선 방향: "통계 탭 / 응답 탭 / 다운로드 탭" 분리, 문항별 차트 렌더 → 이미지 export로 DOCX 삽입. (07 참조)
### 우선순위: P2 / 난이도: 중
