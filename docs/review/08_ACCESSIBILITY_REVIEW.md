# 08. Accessibility Review — 접근성 (WCAG 2.1 AA 관점)

> 대상: 공공 복지관 서비스 → **웹 접근성 준수는 법적·윤리적 의무**(장애인차별금지법, KWCAG 2.2).
> 응답자 다수가 고령·장애인일 수 있어 접근성은 "부가"가 아니라 "핵심 기능".

---

## 8.1 현황 스캔 결과

| 영역 | 관찰 |
|------|------|
| viewport 메타 | ✅ 정상 (`width=device-width, initial-scale=1`) |
| lang 속성 | ✅ `<html lang="ko">` |
| ARIA 속성 사용 | ⚠️ 매우 적음 — 응답 페이지 1개, 빌더 3개, 네비게이터 5개, 모달류 소수. 대부분 컴포넌트 0개 |
| focus 스타일 | 🔶 일부 `:focus-visible`(섹션 버튼·카드 링크·입력) 존재하나 전역 일관성 없음 |
| 시맨틱 마크업 | 🔶 확인 필요 — div/span 기반 버튼·리스트 가능성 |

| 터치 타깃 | ❌ 44×44px 보장 CSS 없음 |

## 8.2 Critical/High 접근성 이슈

### A. 폼 오류·필수 표시가 스크린리더에 전달되지 않음
- 응답 페이지가 `fieldErrors` 상태로 오류를 관리하나 `aria-invalid`, `aria-describedby`, `role="alert"` 연결이 없다. → 시각장애 사용자는 "제출이 안 되는데 왜인지 모름".
- 필수(`required`)가 시각적 표기(별표 등)에만 의존할 가능성 → 스크린리더에 `aria-required` 필요.
- **위험도: High** / 개선: 각 필드에 `aria-required`, 오류 시 `aria-invalid="true"` + 오류 메시지 id를 `aria-describedby`로 연결, 제출 실패 요약을 `role="alert"`/`aria-live="assertive"`로 발화.

### B. 라디오/체크박스 그룹의 시맨틱
- 선택형 문항이 네이티브 `<fieldset>/<legend>` + `<input type=radio/checkbox>`로 구성되는지 확인 필요. div+onClick이면 키보드·스크린리더 접근 불가.
- **위험도: High** / 개선: 네이티브 폼 컨트롤 사용 또는 `role="radiogroup"`+`role="radio"`+키보드 핸들링.

### C. 터치 타깃·글자 크기
- 최소 44×44px 미보장, 본문 12~13px대(`0.78~0.85rem`) 다수. 고령 응답자 실사용 장벽.
- **위험도: Medium** / 개선: 응답자 UI 본문 최소 16px, 인터랙티브 요소 최소 44px, 사용자 확대(폰트 스케일) 허용.

### D. 색 대비 미검증
- 보조 텍스트·placeholder·비활성 상태의 대비가 4.5:1(본문)·3:1(대형) 기준 충족하는지 미확인.
- **위험도: Medium** / 개선: 디자인 토큰 대비 감사, 상태 표시를 색 단독이 아닌 아이콘·텍스트 병행.

### E. 진행·단계 안내 (페이지 모드)
- 섹션 이동 시 포커스 이동·`aria-live`로 "N/M 페이지" 안내가 없으면 스크린리더 사용자가 맥락을 잃음.
- **위험도: Medium** / 개선: 페이지 전환 시 상단 제목으로 포커스 이동 + live region 공지.

### F. 모달 접근성
- `ConfirmModal`/`QrModal`/`ResponseDetailModal`에 focus trap·`role="dialog"`·`aria-modal`·ESC 닫기·열기 전 포커스 복원이 필요. 일부만 aria 사용.
- **위험도: Medium** / 개선: 공통 접근성 모달 컴포넌트로 통일.

---

## 8.3 좋은 점 (계승)
- `lang`·viewport 기본기 확보, 일부 `:focus-visible` 도입, `print` 미디어쿼리(보고서 인쇄 배려).
- 응답 자동저장·명확한 상태 메시지(마감/대기)는 인지 접근성에 긍정적.
