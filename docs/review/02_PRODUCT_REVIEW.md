## 2.1 전체 제품 구조

### 현재 상태
- SPA(React Router 7). 공개 영역(`/surveys`, `/surveys/:id`)과 관리 영역(`/admin/*`)이 라우트로 분리.
- Firebase가 인증(Google) + DB(Firestore) + 호스팅을 전담. 서버(Functions) 없음 — **Fat client / Thin backend** 구조.
- 도메인 로직 대부분이 `src/firebase/surveys.js`(4,483줄)와 3개 대형 페이지에 집중.

### 문제점
- **서버리스인데 서버 검증이 없다**: 응답 유효성·정원·중복 판정이 전부 클라이언트 트랜잭션 + Firestore Rules로만 방어된다. 규칙이 표현할 수 없는 검증(응답 크기, 콘텐츠 유해성, 속도제한)은 사실상 무방비.
- **단일 앱에 4개 제품이 섞여 있다**: 설문조사, 참여신청 접수, 통계 대시보드, 결과보고서 편집기가 한 코드베이스·한 권한모델을 공유. 각각의 요구가 충돌하기 시작하면 변경 비용이 급증.
- 라우트 레벨 가드(`AdminRoute`)와 데이터 레벨 가드(Rules)가 서로 다른 기준으로 판정 → 이중 유지보수.

### 위험도: **High**
### 개선 방향
- Cloud Functions(또는 Cloud Run) 도입으로 **응답 제출·정원 갱신·집계**를 서버 신뢰 경계로 이동. 클라이언트는 UI, 서버는 진실.
- 제품 경계를 명시: `survey`(조사) / `application`(신청) 도메인을 데이터·화면·권한에서 개념적으로 구분(같은 컬렉션이라도 formType로 분기되는 규칙을 문서화).

### 우선순위: P1 (서버 검증) / 구현 난이도: 높음

## 2.2 폼 유형(Form Type) 구조

### 현재 상태
`surveyConstants.js`에 4종 정의 + 기본값 세트:

| formType | 라벨 | 분기 | 정원 | 중복방지 | 명단/처리상태 |
|----------|------|:----:|:----:|:--------:|:-------------:|
| `targeted_survey` | 특정 설문형 | ✅ | ✖ | ✖ | ✖ |
| `general_survey` | 일반 설문형 | ✖ | ✖ | ✖ | ✖ |
| `targeted_participation_application` | 특정 참여신청형 | ✅ | ✅ | ✅ | ✅ |
| `general_application` | 일반 신청형 | ✖ | ✖ | ✅ | ✅ |

### 문제점
- 폼타입이 **7개 boolean 기능 플래그의 프리셋**일 뿐, 이후 개별 토글로 자유 변경 가능 → 조합 폭발. "특정 설문형인데 정원 켜짐" 같은 비정합 상태가 만들어질 수 있고, 이를 검증하는 불변식이 없다.
- 신규 조사 유형(예: 사례관리 사정, 만족도 정기조사, 사업 사전신청) 추가 시 새 formType를 만들지, 기존 플래그를 조합할지 기준이 없음.

### 위험도: **Medium**
### 개선 방향
- 폼타입을 **역량(capability) 프로파일**로 재정의하고, 조합 불변식을 `normalizeSurveyConfiguration`에 명시(예: 신청형만 정원/명단 허용).
- v2에서는 "조사 vs 신청" 최상위 2분류 → 세부 프리셋 형태의 2단계 선택으로 단순화.

### 우선순위: P2 / 난이도: 중

## 2.3 설문 생성 UX (제품 관점)

### 현재 상태
- `admin/surveys/new` → `SurveyBuilderPage`(2,621줄). 폼타입 선택 → 질문/섹션 편집 → 미리보기 → 저장.
- 16종 질문 유형, 질문 블록 프리셋(`questionBlocks.js`), 완성 템플릿(`formTemplates.js`, 1,211줄).

### 문제점
- 빌더가 "모든 것을 한 화면에서" 방식 → 처음 쓰는 직원에게 인지 부담. (상세는 04_UX)
- 저장 payload 생성·검증·정원설정이 한 컴포넌트에 있어 **설문 하나 저장 실패가 전체 화면을 막을 수 있음**.

### 위험도: Medium (제품)·High (UX)
### 개선 방향: 위저드형 단계화 + 실시간 미리보기 (04_UX 참조)
### 우선순위: P1 / 난이도: 중

## 2.4 수정 · 복제 · 템플릿 구조

### 현재 상태
- **수정**: `updateSurvey`가 운영 `questions`를 직접 덮어씀. `draftQuestions` 필드는 생성 시 함께 쓰이나(‌`createSurvey` L2041), 편집 화면이 이를 독립적으로 쓰지 않음.
- **복제**: `duplicateSurvey(surveyId, createdBy)` 존재 — 소유자만 사본 생성.
- **템플릿**: `survey_templates` 컬렉션. 설문 → 템플릿 저장(`buildSurveyTemplateData`), 템플릿 → 설문 인스턴스화(`instantiateSurveyTemplate`), 사용횟수 카운트.

### 문제점 (Critical 후보 — KI-005)
- **수정 즉시 공개 반영**: 운영 중 게시된 설문을 편집·저장하면 응답자 화면이 즉시 바뀐다. 진행 중 응답자의 문항 ID 정합성이 깨지고, 이미 받은 응답과 새 문항 구조가 어긋나 통계가 오염될 수 있다.
- 템플릿 `surveyData`가 스키마 버전 없이 통째로 스냅샷됨 → 앱이 질문 구조를 바꾸면 오래된 템플릿 인스턴스화가 깨질 수 있음.
- 복제 시 응답·정원 카운트가 사본으로 새로 시작하는지의 불변식이 문서화되어 있지 않음(코드상 새 문서라 0으로 시작하나 명시 필요).

### 위험도: **High**
### 개선 방향
1. `draftQuestions`(편집본) ↔ `questions`(게시본) 완전 분리. 편집은 draft만 수정, "게시" 액션에서만 draft→questions 복사 + `version++`.
2. 응답에 `submittedVersion` 저장(스키마에 Future로 이미 예고됨) → 버전별 통계 분리.
3. 템플릿 `surveyData`에 `schemaVersion` 부여, 인스턴스화 시 마이그레이션 훅.

### 우선순위: P1 / 난이도: 높음

## 2.5 확장성 (다양한 조사로의 성장)

### 현재 상태
- 질문 유형·폼타입·정원·분기 등 표현력은 이미 상당. 복지관의 대다수 조사·신청 시나리오를 커버.

### 문제점
- **스키마 버전 필드 부재**: `surveys`·`responses`·`survey_templates` 어디에도 `schemaVersion`이 없음. 데이터 모델이 진화하면 legacy 호환 코드가 계속 쌓임(현재 이미 `normalizeStoredRole`이 8가지 role 문자열을 흡수).
- **집계가 실시간 계산**: 통계는 응답을 전부 읽어 계산(`buildSurveyAnalytics`). 응답 수천 건 조사에서 읽기 비용·지연 증가.
- **다국어/다기관 확장 여지 없음**: 이메일 도메인(`@yeongjung.or.kr`)과 조직명이 코드·규칙에 하드코딩.

### 위험도: **Medium**
### 개선 방향
- 모든 핵심 문서에 `schemaVersion` 추가(12_Migration 참조).
- 응답 시점 증분 집계(집계 문서 또는 Functions) 도입.
- 조직 식별자·도메인을 설정값으로 외부화(멀티테넌시 대비, 필요 시).

### 우선순위: P2 / 난이도: 중
